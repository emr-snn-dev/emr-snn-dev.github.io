###セキュリティ・アップデート概要 (2026-02-08)
本プロジェクトの認証システム更新にあたり、公開リポジトリにおけるセキュリティ・プラクティスに基づいた設計変更を行いました。

🛠 エンジニア向け：秘匿情報の外部化と多層防御
公開Webサイト（GitHub Pages等）において、ソースコードは誰でも閲覧可能な「パブリックな情報」です。

1. 認証ロジックの非公開化
フロントエンドのJavaScriptに「正解（またはそのハッシュ値）」をハードコードすることを廃止する方向へシフトします。

リスク: ハッシュ値がコード上にあると、オフラインでの辞書攻撃（総当たり）が可能です。

対策: 合言葉の検証をクライアントサイドから切り離し、**Firebase側（データベースの読み取り権限やCloud Functions）**へ隠蔽する設計を推奨。ソースコード上からは「正解に関する一切の痕跡」を排除します。

2. 環境変数とバックエンドルールの活用
GitHubリポジトリには「仕組み（コード）」のみを置き、「秘密（鍵）」は以下の場所で管理します。

GitHub Secrets: 自動デプロイ時の環境変数管理。

Firebase Security Rules: ログイン済みユーザーのUIDに基づき、サーバー側でアクセスを強制遮断。

👥 一般（メンバー）向け：安心して利用いただくために
サイトを安全に、かつ便利に使うための「二重の鍵」について解説します。

1. なぜ「合言葉」が必要なのですか？
このサイトはインターネット上に公開されているため、世界中からアクセスが可能です。 まず最初のドア（合言葉）を設けることで、関係のない第三者や、自動で動くプログラム（ボット）がログイン画面をいたずらすることを防いでいます。

2. 二重の鍵で守る「強力なセキュリティ」
もし合言葉を知られたとしても、あなたのデータは守られています。

1つ目の鍵（合言葉）: クラブの入り口の門。

2つ目の鍵（Google/メールログイン）: 各個人の部屋の鍵。 たとえ門を通過できても、本人しか持っていない2つ目の鍵がなければ、活動内容などの重要な情報には一切アクセスできません。

3. 不正アクセスの防止
間違った入力を繰り返すと、システムが自動的にそのブラウザからの操作を一時停止します。これにより、悪意のある攻撃を物理的に防いでいます。
